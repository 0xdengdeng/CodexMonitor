# 企业 AI 开发工作台前端设计规范

本文档定义第一版 `企业 AI 开发工作台` 的桌面 GUI 视觉风格、主题、国际化与组件设计原则。目标用户是国内企业中的项目负责人、业务人员、普通研发协作者，以及少量技术管理员。

## 1. 产品气质

第一版采用「新手友好型企业工作台」风格。

核心气质：

- 可信：第一眼像企业级软件，而不是个人实验工具。
- 清楚：非技术用户能理解当前可以做什么、下一步该做什么。
- 低门槛：降低 AI 编程、代码、终端等概念带来的心理压力。
- 有效率：工作区仍保留开发工具的密度和专业能力。

避免方向：

- 不做纯黑 IDE 风格。
- 不做过度营销化、强渐变、强装饰的 AI 产品风格。
- 不做过轻、过软、缺少企业可信度的个人效率工具风格。
- 不把终端、日志、代码区域的视觉语言扩散到整个应用。

## 2. 颜色系统

第一版采用 P2「标准企业蓝」。

### 2.1 浅色主题

浅色主题是默认主题。

建议 token：

| Token | 用途 | 建议值 |
| --- | --- | --- |
| `color.brand.primary` | 主按钮、主导航选中、关键操作 | `#2563EB` |
| `color.brand.primaryHover` | 主按钮 hover | `#1D4ED8` |
| `color.brand.soft` | 选中背景、轻提示背景 | `#EFF6FF` |
| `color.brand.border` | 品牌浅边框 | `#BFDBFE` |
| `color.bg.page` | 页面背景 | `#F7F9FC` |
| `color.bg.surface` | 面板、表单、卡片背景 | `#FFFFFF` |
| `color.bg.subtle` | 侧栏、次级区域背景 | `#F8FAFC` |
| `color.border.default` | 默认边框 | `#DBE3EF` |
| `color.text.primary` | 主文本 | `#172033` |
| `color.text.secondary` | 次级文本 | `#667085` |
| `color.text.muted` | 弱提示文本 | `#98A2B3` |
| `color.warning.bg` | 警告背景 | `#FFF7ED` |
| `color.warning.text` | 警告文本 | `#9A3412` |
| `color.warning.border` | 警告边框 | `#FED7AA` |
| `color.success.text` | 成功文本 | `#047857` |
| `color.danger.text` | 危险文本 | `#B42318` |

### 2.2 深色主题

保留深色主题，但深色模式应是「企业深蓝灰」，不是纯黑开发者工具。

建议原则：

- 页面背景使用深蓝灰，例如 `#101828`、`#111827` 一类。
- 面板背景略浅于页面背景，例如 `#1D2939`。
- 主色仍使用企业蓝，但降低大面积高亮。
- 终端和代码区域可以更深，但不反向主导全局风格。
- 边框使用低对比度蓝灰，避免刺眼。

## 3. 主题模式

必须保留三种主题模式：

- 浅色
- 深色
- 跟随系统

默认主题：浅色。

用户选择主题后保存到本地配置，重启后保持。登录页、首页、工作区、设置页都必须遵守同一主题规则，不能出现登录前后风格断裂。

## 4. 国际化

第一版必须具备国际化底座，并支持中英文手动切换。

### 4.1 语言范围

必备语言：

- `zh-CN`
- `en-US`

默认语言：

- 首次进入时，如果系统语言是中文或英文，跟随系统语言。
- 如果系统语言不是中文或英文，默认 `zh-CN`。
- 用户手动选择后，以用户选择为准。

### 4.2 入口位置

语言切换入口常驻在全局标题栏右上角。

要求：

- 登录页、首页、工作区、设置页均可切换语言。
- 设置页中保留完整语言设置项。
- 标题栏入口保持轻量，可使用「地球图标 + 当前语言」或 `中文 / EN`。
- 切换语言后立即生效，并保存到本地配置。

### 4.3 文案规则

新增 UI 文案必须走翻译 key，禁止在组件中直接硬编码中文或英文。

需要翻译：

- 登录页
- 首页
- 新手引导
- 空状态
- 菜单
- 按钮
- 设置项
- 表单 label
- toast
- modal
- 错误提示
- 权限提示

不翻译：

- 租户名称
- 用户名称
- 公司名称
- 项目名称
- 模型名称
- 命令
- 代码
- 终端输出
- 日志原文
- API key、token、路径、URL

翻译 key 命名建议按功能域组织，例如：

```txt
auth.login.title
auth.login.apiKeyPlaceholder
settings.language.title
settings.theme.defaultLight
home.newTask.title
workspace.empty.importProject
```

## 5. 布局原则

第一版界面应优先服务非技术用户。

布局原则：

- 首屏直接进入可操作工作台，不做营销落地页。
- 新用户优先看到「连接项目」「描述需求」「确认修改」等清晰动作。
- 工作区可以信息密度较高，但首页、登录、空状态必须更清楚。
- 侧栏导航保持稳定，不频繁移动核心入口。
- 不在页面区块外包裹过多卡片，避免卡片套卡片。
- 表单、设置、任务列表采用企业软件常见的稳定布局。

## 6. 组件风格

组件整体采用克制、清楚、企业化的视觉语言。

基础规则：

- 圆角以 6px 到 8px 为主。
- 按钮、输入框、选择器高度保持稳定，避免文字变化造成跳动。
- 图标优先使用现有图标库，按钮中有合适图标时使用图标增强识别。
- 主按钮使用企业蓝，只用于主要动作。
- 危险动作必须使用明确的危险色和二次确认。
- toast、modal、空状态要给出下一步动作，不只描述问题。

## 7. AI 开发场景表达

面向非技术用户时，避免把底层开发概念直接暴露为主语言。

推荐表达：

- 「新建任务」
- 「导入项目」
- 「确认修改」
- 「查看变更」
- 「等待你确认」
- 「AI 正在处理」

谨慎表达：

- shell
- diff
- patch
- repository
- branch
- sandbox

这些专业词可以在工作区、日志、开发者设置中出现，但首页和新手流程应优先使用业务可理解的表达。

## 8. 与 Codex 官方同步的约束

为了保持后续同步官方更新，前端改造应遵守以下约束：

- 颜色、主题、国际化尽量收敛到 token、配置和翻译资源中。
- 避免在上游高频变化组件中做大面积结构重写。
- 首页、登录、设置等企业定制入口可以独立封装。
- 与 app-server、Codex 核心协议相关的 UI 文案和状态展示应保持兼容，不随意改协议语义。
- 如果必须修改上游组件，优先做小补丁、小 wrapper、小配置，不做整文件替换。

## 9. 第一版验收标准

第一版前端风格完成后，应满足：

- 默认打开是浅色企业蓝风格。
- 深色模式可用，并且不是纯黑 IDE 风格。
- 标题栏右上角可以手动切换中文和英文。
- 用户选择的语言和主题可以保存。
- 登录页、首页、设置页、新建任务入口没有硬编码文案。
- 非技术用户能在首页理解下一步可以做什么。
- 开发者工作区仍保留足够的信息密度和专业能力。
